## Codebase Patterns
- Practical or closing chapter intros can still use the shared `overview` layout; make the three pillars cover the threat model or behavior shift plus the roadmap into actionable subslides.
- Parent chapter intro slides should be added as a group-level `overview` object in `src/data/slides.js`; the shared overview renderer in `src/App.jsx` already maps that shape into the standard chapter layout.
- Child slides can be filled into `src/data/slides.js` incrementally through `childContent`; missing later entries safely render the shared unfinished-slide placeholder until their story is implemented.
- Historical/context subslides can reuse the `ledger` layout when they read cleanly as "before vs proposal" plus a short dated sequence, so simple origin stories do not need custom timeline components.
- Pending-versus-confirmed lifecycle slides also fit the `ledger` layout well: use the two comparison cards for state before and after confirmation, and the side flow for the broadcast-to-inclusion path.
- Capability-comparison slides can also use the `ledger` layout by putting the incumbent system on one card, the expanded system on the other, and the side flow on the reason the newer model exists.
- Consensus comparison slides can also use the `ledger` layout by putting Proof of Work and Proof of Stake in the two cards and using the side flow to narrate how each model advances accepted history.
- Smart-contract concept slides can also use the shared `ledger` layout by comparing centralized backend logic with on-chain code, then using the side flow for the deploy-call-execute-update lifecycle.
- Capability-versus-risk slides can also use the shared `ledger` layout by putting concrete use-cases in one card, failure modes in the other, and using the side flow to show how the same automation creates both leverage and risk.
- Smart-contract execution lifecycle slides can use the shared `pipeline` layout when the story is deploy -> call -> deterministic execution -> state update; that keeps execution explainers data-driven without new React components.
- Protocol-upgrade and ecosystem-coordination slides can use the shared `pipeline` layout when the story is discussion -> EIP/research -> client implementation -> network adoption, which keeps governance explainers sequence-driven without custom React work.
- Structural explainer slides can use the shared `anatomy` layout with `frameLabel`, `segments`, and `notes` when a concept needs its internal parts, roles, or consequences separated visually without adding slide-specific React code.
- State-machine explainer slides can also reuse the shared `anatomy` layout by assigning one segment each to the acting account, the shared state snapshot, and the execution environment that computes the next state.
- Cause-and-effect security slides can use a shared `hash-chain` layout with `chain` and `tamperedChain` arrays in `src/data/slides.js`; the renderer can show an intact path and a broken reference without bespoke slide components.
- Monetary-policy slides can use the shared `issuance` layout with `process`, `eras`, `stats`, and `notes` when one slide needs both a protocol workflow and a stepped reward or supply schedule.
- The shared `issuance` layout can also explain a fixed cap if `eras` are framed as cumulative supply milestones; the same bar treatment reads well for "approaching the ceiling" stories.
- Wallet mental-model slides can reuse the shared `anatomy` layout by splitting the explanation into public-side details, private-key control, and wallet-software responsibilities.
- Acronym or framework slides can also reuse the shared `anatomy` layout if each letter or principle is one segment; keep the grid responsive so 5+ cards wrap cleanly without new React code.
- Principle-to-manifestation slides can also reuse the shared `anatomy` layout by pairing each value with one visible ecosystem or governance choice instead of falling back to abstract slogans.
- Category-tour slides can also reuse the shared `anatomy` layout by making each segment one protocol primitive or function, which works well for scanning DeFi building blocks without custom React.
- Ecosystem role-mapping slides can also reuse the shared `anatomy` layout by assigning each segment to one actor group, its job, and its dependency on the rest of the network.
- Resource-pricing slides can also reuse the shared `anatomy` layout by separating what the protocol meters, why that pricing exists, and what misconception or next-step topic should be kept out of scope.
- EOA and key-management slides can also reuse the shared `anatomy` layout by separating the visible account, the recovery secret, and the signature flow; that keeps wallet education accurate without implying the seed phrase is used directly on-chain.
- Multi-model tradeoff slides can also reuse the shared `anatomy` layout by making each segment one operating model and using the notes to spell out control, recovery, and UX tradeoffs without forcing the story into the two-card `ledger` template.
- Security-assumption tradeoff slides can also reuse the shared `anatomy` layout by making each segment one comparison dimension such as attack cost, honest participation, recovery path, or attack surface, then comparing the consensus models inside each card.
- User-security slides can also reuse the shared `anatomy` layout by making each segment one attack surface cluster such as phishing, dangerous approvals, or compromised environments, which keeps OPSEC content practical without new React code.
- Practical checklist or habit slides can also reuse the shared `anatomy` layout by making each segment one repeatable behavior cluster, which keeps final guidance actionable without collapsing into a dense bullet list.
- Composability-versus-fragility slides can also reuse the shared `ledger` layout by putting protocol-stacking upside in one card, cascading risks and practical limits in the other, and using the side flow to show how dependency chains propagate failure.
- Fee-market slides can also reuse the shared `ledger` layout by contrasting calm versus crowded block space in the two cards and using the side flow to map wallet quote, pending competition, and inclusion priority into the user-facing wait-versus-cost tradeoff.
- Motion follow-ups to an existing explainer slide can often keep the same structured data in `src/data/slides.js` and just switch the slide `layout`, which keeps the static lesson and the animated version synchronized.
